# AI 基础概念

---

## 1. 大语言模型（LLM）

**一句话解释：** LLM 是基于 Transformer 架构、通过海量文本训练的大规模神经网络，能理解自然语言并生成文本，但它没有稳定的记忆和事实校验能力。

**例子：** 向 Claude 提问 "用 Solidity 写一个 ERC20 合约"，它能生成完整的合约代码，包括 name、symbol、decimals 和 transfer 函数。

**常见误区：** LLM 不是数据库。它不知道外部世界的最新状态——比如今天的 ETH 价格、你钱包的余额。它只是基于训练数据和上下文内容生成看起来合理的回答。

---

## 2. 提示词（Prompt）

**一句话解释：** Prompt 是你给 LLM 的输入指令，决定了模型理解任务的方式和输出质量。

**例子：** 对比两种 prompt：
- "写个合约"→ 输出随机
- "用 Solidity 0.8.20 基于 OpenZeppelin v5 写一个 ERC20 合约，包含 name、symbol、decimals 和 permit 功能"→ 输出符合预期

**常见误区：** 很多人认为模型"能猜到我要什么"。实际上，prompt 越模糊，输出越不稳定。需要明确：角色、任务、输出格式、约束条件。例如上述例子，规定了使用版本、需要用到哪些库、需要包含哪些功能等。

---

## 3. 上下文（Context）

**一句话解释：** Context 是模型一次请求能看到的全部信息（系统指令 + 用户输入 + 工具返回 + 历史消息），窗口再大也不意味着模型会完美使用每个细节。

**例子：** 给安全审查 Agent 的 context 包含 "用户地址 0xabc、当前余额 100 USDT、spender 是 Uniswap Router、approve 额度是 1000 USDT"——模型据此判断是否有授权风险。模型只基于上面这堆信息来回答。它不知道：用户昨天新开了一个不认识的 dApp，ETH 价格最近的跌涨等信息。

这就是 Context 的边界：模型只能看到你放进 Context 的东西。没放进去的，它不知道，也不会自己去找。

**常见误区：** "窗口大 = 效果好"是错的。长上下文中模型容易被无关信息分散注意力，关键信息应放在 prompt 的开头和结尾（首位效应）。Memory 不能替代实时授权——用户过去允许的动作，不代表现在仍然允许。

---

## 4. 检索增强生成（RAG）

**一句话解释：** RAG 是让模型在回答前先从外部知识库检索相关文档，把检索结果作为上下文注入，从而避免模型凭空编造。

**例子：** 用户问 "OpenZeppelin 的 Ownable 怎么用？"，系统先检索 OZ 文档找到 Ownable 章节，把内容拼进 prompt，模型再基于文档内容回答。

**常见误区：** RAG 不是万能的。检索质量决定回答质量——如果检索到的 chunk 不相关，模型会基于错误信息编造。需要测试三类问题：文档中有的、文档中没有的（应拒答）、版本不确定的（应提示核对）。

---

## 5. 智能体（Agent）

**一句话解释：** Agent 是能自主调用工具、规划多步任务、管理状态并反思自身输出的 AI 系统。

**例子：** DAO 提案研究 Agent：
1. 接收提案 ID → 2. 检索提案正文 → 3. 分析支持和反对理由 → 4. 标记缺失信息 → 5. 输出投票检查清单

**常见误区：** Agent 不等于"自动执行"。每一步的权限需要隔离——上一步的授权不应自动延续到下一步。Reflection（自我反思）是 Agent 安全的关键：执行 → 检查结果 → 不符合预期则重试或上报。

---

## 6. 框架（Frameworks）

**一句话解释：** 框架（如 LangChain、LangGraph、Agents SDK）提供 tool schema、trace、状态管理和测试工具，帮你组织 Agent 工作流而非从零搭建。

**例子：** 同一任务（OZ 文档问答）：
- 无框架：直接 fetch 文档 → 切 chunk → 拼接 prompt → 输出
- 有框架：定义 tool schema → 自动 trace → 失败重试 → 测试样本

**常见误区：** 框架不是必须的。如果任务简单（单步检索 + 回答），无框架方案更轻量。只有需要多步规划、状态恢复、复杂追踪时才值得引入框架。

---

## 7. 氛围编程（Vibe Coding）

**一句话解释：** Vibe Coding 是用自然语言描述需求，让 AI Coding Agent 完成全部编码工作，人类只负责审核和决策。

**例子：** 给 Claude Code 一个 prompt："给 DAO Agent 添加 --compare 功能，对比两个提案"，它自动读代码 → 出计划 → 写 ~150 行代码 → 跑 7 个测试 → 输出 PR summary。

**常见误区：** AI 写的代码仍需人工审查。常见问题包括：prompt 中没定义的输出格式（AI 自行设计）、缺少边界情况的单元测试、涉及安全/资金的逻辑需要人类判断。AI 是加速器，不是替代审阅者。

---

## 8. 模型上下文协议（MCP）

**一句话解释：** MCP 是 LLM 与外部工具/数据源之间的标准通信协议（类似 USB 标准化外设连接），Server 暴露工具，Client 发现并调用。

**例子：** 你想让 LLM（比如 Claude Code / Hermes）能读取你本地的文件。但不可能把整个文件系统塞进 context，也不可能让 LLM 直接读任意文件（太危险）。
这时就需要写一个 MCP Server，只读 MCP Server 暴露两个工具：
- `search_docs(query)` — 只允许在白名单目录中搜索
- `get_file(path)` — 只允许读取白名单目录内的文件
每次调用写日志，路径遍历攻击被拒绝返回 `{ error: "..."}`。

**常见误区：** MCP 不是安全万能的。即使 Server 有权限控制，Client 端仍可能滥用工具。写入操作需要额外确认机制（如一次性 token + 60s 过期 + 审计日志）。

---

## 9. 评估（Evaluation）

**一句话解释：** Evaluation 是通过预设的测试样本（正常/边界/失败/恶意）系统性地衡量模型输出质量的流程。

**例子：** 你做了个 RAG 系统，怎么知道它好不好？跑几条"正常问题"感觉还行，但一上线就可能被用户发现各种问题。这时就需要评估集，RAG 评估集 30 条样本设计逻辑：
- 10 条正常问题（文档中明确有答案）———测"能不能用"- 10 条边界问题（版本兼容性、概念对比）———测"会不会偏"- 5 条预期失败（不存在的 API）———测"该拒绝的是否拒绝"- 5 条恶意注入（prompt injection）———测"安不安全"**常见误区：** 评估不是一次性工作。每次改 prompt、模型或检索策略后都要重新跑一遍。注意只测正常用例就相当于没有评估，边界和恶意用例才是发现问题的关键。

---

## 10. 微调（Fine-tuning）

**一句话解释：** Fine-tuning 是在预训练模型基础上用领域数据继续训练，让模型掌握特定领域的输出模式和知识。

**例子：** 有时候你怎么改 prompt，模型就是做不好某件事。比如你让它从一份技术文档中提取风险点，不管怎么修改 prompt，它把字段全部输出，但风险点一条都没抓到或捕捉不全。这时候就需要微调。

**常见误区：** 微调不是第一步，而是最后一步。先试 prompt 工程和 few-shot——如果这两种方案能稳定解决问题，就不需要微调。微调的成本（数据标注 + GPU + 评估）远高于 prompt 调优。

---

## 11. 推理服务（Inference）

**一句话解释：** Inference 是把训练好的模型部署为在线服务，在延迟、成本、质量和隐私之间做取舍的工程决策。

**例子：** 你写了一个 AI 功能，比如"交易风险分析器"。现在你要决定：这个功能跑在什么上面？（是让用户每次请求都去调用远程的 GPT/Claude API？还是在自己的电脑上跑一个轻量模型？或者干脆写几条 if-else 规则？）

这就是 Inference——模型部署方式的工程决策。

**常见误区：** "模型越大越好"是错的。对简单规则类任务（如检查地址是否为合约），本地规则引擎更快、更便宜、更隐私。API 模型擅长复杂推理但延迟高、有成本、数据需外发。混合方案才是产品级选择。
