**核心要点：**

### 🪟 Context Window
- 模型一次请求能处理的最大上下文范围
- 窗口大 ≠ 模型会用得好，"看见了但没抓住重点" 是常见问题
- 在实践中要配合检索、摘要和结构化数据使用

### 🏗️ Context Engineering
- 设计上下文进入模型的方式：数据源选择、排序、裁剪、标注来源
- 一个稳定的工具型 Agent 上下文应包括：系统指令、当前状态、工具返回、用户偏好、外部数据、历史摘要
- 不可信内容要隔离标注

### 🧠 Memory
- 跨请求保留的信息（用户偏好、历史任务、常用地址）
- 方便但有隐性风险——Memory 不能替代实时授权
- 用户过去允许的动作，不代表现在仍然允许

### 📚 Knowledge Base
- 系统可检索的外部知识库（文档、FAQ、runbook）
- 与 Memory 的区别：KB 是公开知识，Memory 是用户定制

## 实践成果

### 🔧 LLM 最小实践：交易解释器
- 使用 Claude Code + ethers.js v6 创建链上交易解析脚本
- 通过 Sepolia 测试网 RPC 读取交易详情，输出格式化表格和事件日志
- 验证交易哈希：`0x711b963df8a432c8f24580681de76b6d7294ac6cfb930ba0d8f2a15cf22ee9fc`
- 将链上数据喂给 Claude 做自然语言解释，成功输出：
  - 交易摘要
  - 链上事实（from / to / value / 事件）
  - 模型推断
  - 不确定性标注
  - 用户检查清单

**练习要点验证：** 实现了 Handbook 要求的"把模型生成、链上事实、来源边界、不确定性分开"

### 🔧 Context 最小实践：钱包授权检查 Agent
- 设计了一份完整的 context spec，覆盖 5 个维度：
  1. 必须实时查询的字段（allowance / balance / gas / spender 合约代码）
  2. 建议缓存的字段（合约标签 / 恶意地址库 / TTL 策略）
  3. 用户提供的上下文（意图 / 钱包类型 / 操作场景）
  4. 外部不可信内容（dApp 声明 / 域名 / 社交媒体链接 → 隔离标注）
  5. 模型绝对不能当成事实的字段（合约名≠安全、无限授权≠常见即安全）
- 代码位置：`experiments/context-spec.md`

### 🔧 RAG 最小实践：协议文档 RAG 问答
- 用 Claude Code 搭建完整 RAG 系统：scraper → retriever → LLM 问答
- 知识库：OpenZeppelin Contracts 5.x 文档
- 技术架构：fetch 爬取 10-15 页面 → 按标题切 chunk → TF-IDF 检索 Top 5
- 三类测试结果：

| 测试类型 | 问题 | 结果 |
|---------|------|------|
| ✅ 文档中存在 | "How to create ERC20 with Ownable?" | 检索命中 90 chunks，Top 5 按相关度排序 |
| ✅ 不存在的问题 | "What is the gas price on Polygon?" | 成功拒答：文档不包含此信息 |
| ✅ 版本依赖问题 | "Access control in Solidity 0.6?" | 正确提示：v5.x 最低 0.8.20，Sol 0.6 不支持 |

- 输出格式完整：Answer / Source / Uncertainty / Needs Version Check
- 代码位置：`experiments/rag-mininal-practice/`
