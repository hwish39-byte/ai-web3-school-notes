**核心要点：**

### ️ Server
- MCP Server 提供工具和资源，通过标准协议暴露给 Client
- 可运行在本地或远程

### Client
- MCP Client 是消费端（如 Hermes、Claude Code）
- Client 发现 Server 的工具列表 → 调用工具 → 处理返回

### Tool Schema
- 每个工具需要定义：名称、描述、参数（JSON Schema）
- Client 根据 Schema 决定何时调用、传什么参数

### Permission
- 工具调用权限控制：白名单、用户确认、撤销机制
- 写入操作需要比只读操作更严格的权限

## 实践成果

### MCP 最小实践：只读 MCP Server
- 用 Claude Code 创建零依赖 MCP Server，HTTP 端口 3100
- 两个工具：`search_docs(query)` + `get_file(path)`
- 安全措施：路径遍历双重检查、扩展名白名单、零静默失败
- 6 项测试全部通过
- 内置写入工具权限升级方案：一次性 token（60s 过期）、撤销 API、SHA-256 审计日志
- 代码位置：`experiments/mcp-mininal-practice/`

### Evaluation 最小实践：RAG 评估集
- 为 RAG 问答系统创建了 30 条 eval 样本：
 - normal: 10 条 / boundary: 10 条 / failure: 5 条 / malicious: 5 条 
- 每条样本包含：输入 / 期望行为 / 必须包含信息 / 必须拒绝情况 / 是否需要引用
- 代码位置：`experiments/eval-mininal-practice/eval-suite.json`

### Fine-tuning 最小实践：结构化摘要微调前评估
### Fine-tuning 最小实践：结构化摘要微调前评估
- 准备 50 条结构化摘要样本（summary / risks / open_questions / sources）
- 比较三种方案（Prompt Only / Few-Shot / LoRA 设计）
- 发现：两种 prompt 方案均无法解决风险识别问题（100% 遗漏风险点）
- 方案 C 设计了 LoRA 方案：$2,630 成本估算
- 代码位置：`experiments/fine-tuning-mininal-practice/`

### Inference 最小实践：交易风险推理对比
- 比较规则引擎 vs API 模型的交易风险评估

| 交易场景 | 规则引擎 | 延迟 | API 模型 | 延迟 |
|---------|---------|------|---------|------|
| ETH 转账 0.5 ETH | LOW | 1ms | LOW | 26.6s |
| ERC20 USDT 转账 | MEDIUM | <1ms | LOW | 21.1s |
| DEX Swap 15 ETH | HIGH（误报） | 1ms | LOW | 25.3s |

- 关键发现：规则引擎在 DEX swap 上因 unlimited approval 误报高风险，API 模型能正确识别 DeFi 标准行为
- 设计了三层 fallback 方案：规则引擎初筛 → 模糊场景 fallback API → 人工审核兜底
- 代码位置：`experiments/inference-mininal-practice/`
