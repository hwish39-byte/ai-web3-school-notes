**核心要点：**

### 🛠️ Tool Use（工具调用）
- Agent 从"回答问题"到"执行动作"的关键能力
- 模型需要知道：有什么工具、工具参数、什么时候该用、返回结果怎么处理
- 你已经在用 Hermes 的 tools 了——这就是 Tool Use 的实际体现

### 📋 Planning（任务规划）
- Agent 把复杂任务拆成多步计划，按序执行
- 关键：计划可验证、可回滚、有终止条件
- 你昨天做 RAG 的 scraper → retriever → LLM 就是简单的 planning

### 🔄 State（状态管理）
- Agent 在每一步需要知道当前状态（已完成什么、下一步是什么）
- 状态泄漏 = 安全风险（上一步的权限不应自动延续到下一步）

### 🔍 Reflection（自我反思）
- Agent 在执行后检查自己的输出是否合理
- 关键模式：执行 → 检查结果 → 不符合预期则重试或上报

### 👥 Multi-Agent（多智能体）
- 多个 Agent 分工协作：一个做研究、一个做分析、一个做执行
- 关键：Agent 之间的通信协议和权限隔离

## 实践成果

### 🔧 Agent 最小实践：DAO 提案研究 Agent
- 用 Claude Code 创建提案研究 Agent，支持三种调用方式：
  - `node dao-agent.mjs uniswap-1` — 提案 ID 查询
  - `node dao-agent.mjs https://boardroom.io/proposals/aave/3` — URL 直接读取
  - `node dao-agent.mjs arbitrum-2 --output report.md` — 输出到文件
- Agent 功能：读取提案正文 → 分析支持/反对理由 → 标记缺失信息 → 输出投票检查清单
- 严格保持只读，不执行任何交易
- 代码内含"权限升级版本"设计方案：用户授权 + simulation 后才可生成投票交易草稿
- 代码位置：`experiments/dao-agent.mjs`

### 🔧 Frameworks 最小实践：框架对比
- 同一任务（OZ Ownable 文档问答），用两种方式实现：
  - **direct.mjs** — 无框架，直接检索 + 拼接 prompt → 输出答案
  - **framework-style.mjs** — 带 tool schema、trace 日志、失败重试、测试样本
- 主要差异：direct 版输出答案直接简洁；framework 版多了 trace 日志、检索延迟、工具调用记录
- 对比维度：可读性、加工具难度、定位错误难度、回归测试难度
- 代码位置：`experiments/framework-compare/`
