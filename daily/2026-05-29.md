### 机器支付（Machine Payment）核心要点

- Agent 如果只能免费调用工具，能力就停在 demo 阶段。真实服务需要付费：模型推理、数据 API、浏览器环境、链上交易
- 机器支付的挑战不是转账本身，而是：**谁授权、价格多少、什么时候扣款、失败怎么退、收据如何验证、预算如何不被滥用**
- **Stablecoin Payment**：稳定币适合机器支付（价格稳定、结算快、可编程），但 Agent 必须区分计价币种和结算币种（0.1 ≠ 0.1 USDC ≠ 0.1 gas token）
- **Budget**：用户授权给 Agent 的最大支出范围，应按时间/任务/服务方/币种/额度分层。全局预算 → 任务预算 → 单次上限 → 服务方上限 → 紧急停止
- **Quote**：服务方给 Agent 的可执行报价，包含服务描述、价格、有效期、交付条件和退款条款
- **Payment Intent**：将"付款意图"封装为结构化对象，包含付款方、收款方、金额、链、token、引用 ID、预算来源和过期时间
- **x402**：HTTP 402 Payment Required 的链上版本，允许服务方在 API 响应时附带付款要求和报价
- **MPP / Subscription / Micropayment**：多边支付、订阅模型和微支付场景

### 结算与托管（Settlement & Escrow）核心要点

- 解决 Agent 经济里的核心矛盾：**付款和交付之间有时间差**——先付钱可能被骗，先交付可能收不到钱
- **Escrow**：资金先锁定在合约或可信账户，等待交付条件满足后再释放。关键设计是**状态机**：Created → Funded → Delivered → Accepted → Refunded / Disputed → Released
- **核心概念**：
  - **Receipt**（收据）：链上交付和付款的不可篡改凭证
  - **Delivery Proof**（交付证明）：结果、hash、日志、模型输出、交易哈希都可能成为证据
  - **Acceptance**（验收）：谁来确认服务完成、有何标准、可接受误差范围
  - **Refund**（退款）：部分失败、超时未交付、服务不符预期时的退款规则
  - **Dispute**（争议）：仲裁机制和评判方选择
  - **ERC-8183**：参考标准，定义 Agent Escrow 合约的最小接口
- 第一性原理：**如果任务定义模糊，Agent 和服务方就无法判断何时完成、谁来确认、失败怎么办。Escrow 设计首先要定义状态机，而不是先写付款代码。**

### 第一性原理

支付是执行能力的一部分。一旦 Agent 可以付款，它就可以消耗用户资金、触发链上状态变化、购买外部服务或被恶意上下文诱导。结算的核心不是"把钱打过去"，而是把任务、交付、验收和付款绑定成可验证流程。**预算、报价、收据、争议处理——这些不是付款的附件，而是支付系统的基础组件。**
