### 核心要点

#### Chain-aware Context（链感知上下文）

- **第一性原理**：模型不能凭语言记忆判断链上事实，链上事实必须从工具和索引层读取
- **On-chain Data**：读取时要带 chain id、block number、contract address、method、返回值、读取时间
- **Contract Docs**：ABI 只能提供函数签名，不能解释业务语义；需要 NatSpec、README、审计报告补充
- **ABI / Event**：能调用 ≠ 应该调用；写交易前需要权限、余额、allowance、slippage、simulation 检查
- **Transaction History**：保留 tx hash、block number、from、to、method、value、token transfers、logs
- **Explorer Context**：给出 explorer link 比只给一句"交易成功"更可靠
- **Indexing Context**：索引上下文必须带时间戳和同步状态；落后 500 个区块的数据不能当当前事实
- **Citation**：没有 citation 的链上解释只能算观点；带 citation 的解释才有机会被验证和追责

#### Web3 工具调用（Web3 Tool Use）

- **第一性原理**：模型可以选择工具，但工具必须用确定性边界限制模型
- **读写分离**：读取链上状态和发送交易必须是不同工具、不同权限
- **RPC Tool**（初级）：只读 RPC 可开放更宽，写入能力要拆出去
- **Contract Read**（初级）：最常用也相对低风险的 Web3 工具
- **Contract Write**（高级）：写交易前需要模拟、policy 检查、用户授权、交易追踪
- **Wallet Tool**（高级）：最敏感的边界，必须把连接、签名、发送交易、授权、撤销分别处理
- **DeFi Tool**（高级）：直接影响资产，需要协议白名单、最大交易额、滑点、simulation
- **Tool Permission**（高级）：按工具、合约、方法、金额、时间、频率分层
- **Tool Log**（中级）：每次工具调用记录用户目标、输入、输出、错误、时间、链 ID、交易哈希

## 学习路径

### 最小路径（半小时内能完成的入门任务）
1. 阅读 Chain-aware Context 章节内容（约 5 分钟阅读）
2. 找一笔公开交易哈希，收集 chain id、block number、from、to、method、value
3. 写一段模型可读的上下文包，标出哪些是链上事实，哪些是解释
4. 在 WCB Learning 页面登录并记录今日进度

### 推荐路径（正常学习节奏）
1. ✅ 完成最小路径全部内容
2. 阅读 Web3 Tool Use 章节内容
3. 用 ethers.js / viem 写一个只读 RPC 工具：读取某地址在某链上的 ETH 余额
4. 写一个合约读取工具：读取 ERC-20 `allowance(owner, spender)`
5. 写一个交易草稿工具：生成 ERC-20 `approve` calldata（不发送）
6. 为每个工具定义输入 schema、输出字段、错误类型和日志字段

### 挑战路径（有余力时深入）
1. ✅ 完成推荐路径全部内容
2. 选择一个 Hackathon 方向（Agentic Commerce / Dev Tooling / AI Security / Governance）
3. 拆解一个具体场景的「谁发起任务、谁执行、谁付款、谁验证、谁承担风险」
4. 用 Hermes Agent 或编程实现一个完整的 Web3 工具调用链
5. 形成 Week 2 的 Project Proposal 初稿
6. 提交到 Handbook feedback：对 Chain-aware Context / Web3 Tool Use 的补充或修正建议
