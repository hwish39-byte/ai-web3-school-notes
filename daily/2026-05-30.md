### 智能体身份（Agent Identity）核心要点

- Agent Identity 不是给 Agent 起个名字，而是让用户、服务和其他 Agent 能验证：**它是谁、谁控制它、能提供什么能力、服务入口在哪里、历史记录能不能追溯**
- 没有身份，Agent 之间无法建立长期信任：今天的"高质量 Agent"明天可能换了服务端、换了 owner、换了能力描述，用户却无法判断
- **核心**：把 Agent 从临时会话变成可发现、可验证、可追责的经济参与者
- **Agent Profile**：公开说明文件，包含名称、描述、服务范围、价格、接口、钱包地址、能力列表、模型/工具说明、隐私政策、owner 和版本。要同时给人和机器看
- **Capability**：Agent 能完成什么任务，以及需要哪些输入和权限。越具体越有用——"总结治理提案"≠"生成 Solidity 测试"≠"执行小额稳定币支付"
  - 能力如果会触发链上动作，要标记风险等级（只读 = 低风险，生成交易草稿 = 中风险，自动执行交易 = 高风险）
- **Service Endpoint**：外部系统调用 Agent 的入口（HTTPS API、A2A endpoint、MCP server、Webhook 或链上 registry）
  - 安全性直接影响身份可信度；更新需要 owner 签名并保留历史
- **Registry**：Agent 身份的公开目录，支持查找、验证、发现。可以是链上合约、IPFS 索引、DNS-based 或其他去中心化方案
- **DID / Verifiable Credential**：去中心化身份标准和可验证凭证标准，是构建 Agent 身份的技术基础

### 智能体信任与声誉（Agent Trust & Reputation）核心要点

- 解决的核心问题：**当一个 Agent 声称自己能完成任务时，用户和其他 Agent 如何判断它是否可靠、历史是否真实、失败是否有代价**
- **第一性原理**：信任不是一个分数，而是一组可追溯、可比较、可解释的证据
  - **身份先于声誉**：没有稳定身份，历史记录无法积累
  - **信号先于分数**：泛泛五星评价不如具体任务结果
  - **Stake 先于承诺**：没有成本的承诺很容易被滥用
- **Reputation**：Agent 历史表现形成的信号集合（成功率、响应时间、争议率、退款率、用户评分等）。
  - 不要压成一个黑盒分数，最好按任务类型拆开
  - 考虑时间衰减——两年前的好表现不保证今天的服务质量
- **Review**：用户/客户对任务结果的反馈。应绑定任务 ID、交付物、付款记录和评价者身份。
  - 质量 > 数量：一个包含任务说明、验收标准、交付 hash 的 review 比十个 "good job" 更有价值
  - 防止互刷：评价者本身也需要身份和声誉
- **Attestation**：某主体对 Agent、任务或结果做出的可验证声明。应结构化包含 issuer、subject、claim、evidence、expiration、revocation
- **Stake**：Agent 或运营方押上的经济保证——让承诺有成本，失败时有资金可被罚没或赔付
- **Slashing**：验证失败、数据造假、行为违规时，stake 被部分或全部罚没
- **Arbitration**：争议处理机制，决定是否 slashing 以及如何分配罚没资金

### 第一性原理

> Agent Identity 不是给 Agent 起个名字，而是让用户、服务和其他 Agent 能验证身份、能力和历史。
>
> 信任不是一个分数，而是一组可追溯、可比较、可解释的证据。

身份先于声誉——没有稳定身份，历史记录无法积累。声誉先于支付——没有可靠声誉，Agent 经济无法信任。**Agent 身份 + 声誉系统，是 AI × Web3 从 demo 走向真实经济运行的基础设施层。**

## 最小路径（30 分钟内）

1. 阅读本 daily note 的 Agent Identity 和 Agent Trust & Reputation 要点总结（约 10 分钟）
2. 思考一个问题：一个你信任的 AI Agent 应该具备哪些条件？它需要告诉你是谁运营它、它能做什么、历史表现如何吗？
3. 在纸上或笔记里画一个简单的 Agent Card：包含 name、owner、capabilities、service endpoint、reputation 三个核心字段
