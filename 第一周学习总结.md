## 二、每周核心收获

### 1. 从"用 AI"到"理解 AI"

第一天的我对 AI 还是"装好了但没用过"的状态。一周结束后，我不但理解了 LLM 的内部机制（Token、Transformer、Hallucination），还亲手搭建了 RAG 系统、MCP Server、Agent 工作流。最深的体会是：**理解 AI 的能力边界比学会某个工具更重要——知道它擅长什么、不擅长什么、会在哪里出错，才能真正用好它。

### 2. Context 是一切的关键

Context 章节对我触动最大。"模型只能看到你放进 Context 的东西，没放进去的它不知道也不会自己去找"——这句话解释了之前很多 AI 应用"看起来不够聪明"的根本原因。在设计钱包授权检查 Agent 的 Context Spec 时，我真正理解了 Context Engineering 不是技术细节，而是**信息治理**。

### 3. AI × Web3 的安全边界

这一周反复出现的主题：**AI 可以辅助，但不能替代人类决策**。从 Context 最小实践（钱包授权 Agent）到 Inference 推理对比（规则引擎误报 DEX swap），再到受限 Web3 助手设计，我越来越清楚地看到 AI 和链上操作之间的那条线：

- AI 可以读链上数据、模拟交易、生成风险报告
- 但签名、授权、最终决策必须由人来完成
- Session Key + 账户抽象是最有前景的"中间方案"

### 4. 最小实践的学习方式非常有效

Handbook 的"最小实践"设计很巧妙——每个章节都在概念之后配一个可动手的练习。相比于只看不练，这种"读一章做一章"的方式让概念真正扎下根。比如 RAG 章节，如果只看理论可能觉得很简单，但实际搭建时才发现 chunk 切分策略、检索质量、拒答逻辑都是要反复调优的。

### 5. 学习工具本身的体会

用 Hermes Agent 来学习 AI，用 Claude Code 来完成实践——这种"用 AI 学 AI"的方式本身就是一种元学习。过程中我也更清楚地看到了 AI 的优势（快速生成代码、解释概念、梳理结构）和局限（需要人工审阅、有时会编造、不能替代领域知识）。

## 三、一些感受

从最开始连 `git` 指令都要问ai，到一周后能写出 MCP Server、部署合约到测试网、设计 AI × Web3 工作流——这 5 天的密度确实很大。中间有几次卡住的时候（比如 RPC 连接失败、合约 increment 调用报错），但每次解决后对链上操作的理解就更深一层。

之前主要接触的是 Web3 方面的内容。这一周补上了 AI 的基础认知，开始看到两个领域交叉时的具体问题：Agent 怎么理解链上状态、钱包权限怎么和 AI 工作流结合、Session Key 如何让 Agent 在受限范围内自主操作。

下一步希望把 AI × Web3 Bridge 的章节啃完，然后尝试做一个能跑通实际链上操作的 Agent 工具。
