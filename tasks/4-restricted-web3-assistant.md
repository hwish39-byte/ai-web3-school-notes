  1|# 任务：设计一个受限 Web3 助手 / 小 workflow
  2| 3|## 目标
  4|设计一个受限 Web3 助手 workflow——AI 可以辅助规划、解释、生成草稿、检查结果，但不能自动接触私钥、助记词，不能绕过人工确认完成签名、授权、转账或合约写入。
  5| 6|---
  7| 8|## 设计方案：Agent 安全授权检查 Workflow
  9| 10|### 场景
 11|用户在 dApp 上被要求 approve 一笔 token，不确定是否安全，向 Agent 求助。
 12| 13|---
 14| 15|### 1. 用户输入是什么
 16|- dApp 的 URL 或域名
 17|- 钱包地址
 18|- 弹窗显示的 approve 详情（spender 地址、token 合约、额度）
 19| 20|### 2. AI / Agent 做什么
 21|```
 22|AI 可以做的（自动化）：
 23|├── 读取 spender 地址的链上信息（是否已验证合约？部署时间？）
 24|├── 查询用户对该 spender 的历史授权记录
 25|├── 检查 spender 是否在已知恶意地址库中
 26|├── 生成交易模拟结果（模拟如果签名了会怎样）
 27|├── 输出结构化的风险分析报告
 28|└── 生成交易的 EIP-712 解释（"这笔 approve 会把什么权限给谁"）
 29| 30|AI 不能做的（必须人工）：
 31|├── 自动连接钱包
 32|├── 自动签名交易
 33|├── 自动发送交易
 34|├── 接触私钥或助记词
 35|└── 替用户做最终决策
 36|```
 37| 38|### 3. Web3 工具或链上步骤
 39|```
 40|步骤 工具/API AI 参与
 41|━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 42|1. 查 spender 地址 Etherscan API / RPC 自动 
 43| 是否合约
 44|2. 查合约验证状态 Etherscan API 自动 
 45|3. 查用户 allowance RPC (eth_call) 自动 
 46|4. 查用户余额  RPC (eth_call) 自动 
 47|5. 查恶意地址库  本地/链上黑名单 自动 
 48|6. 模拟交易 Tenderly / ETH_SIMULATE  AI 发起，结果返回
 49|7. 生成风险报告  LLM 分析所有数据 自动 
 50|8. 展示给用户  ~~~~ 人工确认线 ~~~~ 停
 51|9. 用户决定是否签名 MetaMask 人工 
 52|10. 记录结果  写入 daily note 自动 
 53|```
 54| 55|### 4. 哪些步骤必须人工确认
 56|```
 57|1.  确认 dApp 域名是否正确（防钓鱼）
 58|2.  确认 approve 的额度和 spender 地址与 AI 报告一致
 59|3.  确认 MetaMask 弹窗里的内容与 AI 预期一致
 60|4.  最终签名——永远由用户手指完成
 61|```
 62| 63|### 5. 如何验证执行结果
 64|```
 65|- 交易模拟结果与预期对比
 66|- 签名后到 Etherscan 查看交易状态
 67|- 检查用户 allowance 是否已更新
 68|- 将交易哈希记录到 daily note 备查
 69|```
 70| 71|### 6. 风险和限制
 72|```
 73|风险：
 74|├── AI 可能误判合约安全性（如未发现隐藏的恶意代码）
 75|├── 钓鱼网站可能绕过 URL 检查
 76|├── 恶意地址库更新不及时
 77|└── 用户可能无视 AI 警告仍然签名
 78| 79|限制：
 80|├── AI 不能访问 MetaMask 内部状态
 81|├── AI 不能拦截或阻止用户签名（只能提醒）
 82|├── 依赖 RPC 和 API 的可用性
 83|└── 无法在离线环境中工作
 84|```
