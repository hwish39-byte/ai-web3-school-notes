# Module B - Payment / Commerce / Settlement

---

## 任务一：场景拆解

### 选定场景：AI 合约审计助手

用户（项目方）提交一份 Solidity 合约源码，Agent 审计并输出风险报告后自动收款。

### 完整流程

| 角色 | 职责 |
|------|------|
| 用户（项目方） | 提交合约、支付审计费 |
| Agent（审计服务） | 读取合约、分析风险、生成报告 |
| 验收者 | 用户 / 自动规则引擎 |
| 仲裁者 | 平台或 DAO（争议时介入） |

1. 下单：用户提交合约源码 + 审计要求（深度/范围/截止时间）
2. 报价：Agent 根据合约复杂度、行数、导入库数量自动报价
3. 预算授权：用户在 CAW Pact 中设定预算上限（如 0.1 ETH）
4. 预存/托管：资金进入 escrow 合约
5. 执行审计：Agent 分析合约，输出结构化风险报告
6. 交付：报告上传 IPFS，返回 CID 给用户
7. 自动验收：规则引擎检查报告是否包含所有必填字段
   - summary / risks / open_questions / sources 完整
   - 有模拟交易记录
   - 来源可追溯到实际代码
8. 付款：验收通过，escrow 释放资金给 Agent
9. 记录：链上记录交付 CID、付款哈希、验收结果
10. 争议：用户可发起争议，人工仲裁

### 谁仲裁

初期采用自动验收（字段完整性 + 基础规则检查），用户如果对结果不满可发起争议，由平台或指定仲裁者人工审核。仲裁结果上链记录。

## 任务二：最小 Payment / Commerce Flow

### 流程

```
用户 提交合约 + 审计要求     -> Agent
Agent 返回报价 + 交付标准    -> 用户
用户 授权预算 (CAW Pact)    -> 资金进入 escrow
Agent 执行审计              -> 分析源码 + 生成报告 + 上传 IPFS
Agent 交付报告 (CID)        -> 用户
验收 (自动规则检查)
  - 字段完整？
  - 来源可追溯？
  - 关键风险已覆盖？
  -> 通过 -> escrow 释放
  -> 不通过 -> 退款 / 争议
链上记录收据
  - 交易哈希
  - 交付 CID
  - 验收结果
```

### 边界条件

| 场景 | 处理方式 |
|------|---------|
| 交付超时 | Agent 未在截止时间内完成，全额退款 |
| 报告不完整 | 缺少必填字段，退回修改，超 2 次退款 |
| 用户不确认 | 超 48 小时未确认，自动视为验收通过 |
| 争议 | 人工仲裁，仲裁结果上链，费用由败诉方承担 |

## 任务三：协议比较

### x402 vs MPP

| 维度 | x402 | MPP |
|------|------|-----|
| 一句话 | HTTP 402 的链上实现：请求被拒，付款，拿到内容 | Agent 间机器支付的通用协议栈 |
| 解决哪段 | 支付入口（付款，解锁资源） | 报价、付款、交付、验证、结算完整链路 |
| 支付方式 | 单次 micropayment | 支持 escrow、分期、订阅、按量 |
| 验证 | 付款即验证（付款成功 = 有权访问） | 需要显式交付验证和验收逻辑 |
| 争议处理 | 无，付款即完成 | 支持 escrow + dispute resolution |
| 适合场景 | 轻量级 paywall（付费 API、付费 AI 推理） | 复杂 commerce（Agent 服务、数据市场） |
| 共同点 | 都解决 agent 自主支付问题，不需要人工确认 |

### 对照说明

x402 解决的是"能不能付"——让 Agent 看到 paywall 后自动识别价格、完成支付、拿到结果。它不关心交付质量和后续争议。

MPP 解决的是"付了之后怎么办"——报价怎么确认、交付怎么验收、失败了怎么退款、有争议谁仲裁。它假设付款只是 commerce 链路的开始，不是结束。

在合约审计助手的场景里：
- x402 适合最小版本：用户付费，拿到报告，不保证质量
- MPP 适合生产版本：托管，执行，验收，付款/退款，有完整链路
